## 문제: 트래픽 10배에 프레임 시간 23배

평상시(50건/초)에는 프레임이 16.7ms안에 그려지지만,
부하를 10배(500건/초)로 올리면 프레임당 340~390ms, 약 2.7fps로 감소

Performance 측정 결과 병목은 Painting(187ms)이 아니라 **Scripting(3,292ms / 3.87s, 85%)**

| 지표           | 평소 (50/s) | ×10 (500/s)         |
| -------------- | ----------- | ------------------- |
| 프레임 시간    | ≤16.7ms     | 340~390ms (약 23배) |
| Scripting 비율 | 47%         | 85%                 |
| DOM 노드       | 17,581 ↑    | 63,268 ↑            |
| JS heap        | 164MB ↑     | 238MB ↑             |

> 측정 조건: Chrome DevTools Performance, 각 3~5초 녹화, 동일 조건에서 비교

- DOM 노드와 heap이 평상시에도 계속 증가
- 로그 1건이 DOM 노드 7개를 만들고 배열은 계속 증가하기 때문에, 부하가 없어도 시간이 지나면 느려지는 건 동일하게 발생

#### 파악된 문제

- 이벤트마다 setState 호출
- 전체 로그를 매번 DOM에 렌더
- 매 렌더마다 전체 배열 순회(집계·reverse)

---

## 개선 1: 이벤트당 setState → 100ms 배치 flush

기존에는 이벤트가 올 때마다 setState로 전체 배열(N건)을 복사
복사 횟수가 유입 속도에 그대로 비례해서, ×10 부하면 초당 500번의 전체 복사가 발생
이벤트를 useRef 큐에 push로 모아뒀다가(복사 없음, O(1)),
100ms마다 한 번에 커밋하도록 변경
전체 복사는 이제 **초당 최대 10회로 고정**

### 결과

| 지표                  | 개선 전                    | 개선 후                   |
| --------------------- | -------------------------- | ------------------------- |
| Scripting 비율 (평소) | 47%                        | **31%** (1,671ms / 5.41s) |
| CPU 패턴 (평소)       | 연속 포화                  | 100ms 주기 펄스           |
| Long Task (평소)      | 상시 발생                  | 거의 소멸                 |
| 프레임 시간 (×10)     | 340~390ms (약 9천 건 시점) | 540ms (약 1.5만 건 시점)  |

- 프레임 시간은 개선되지 않음
  이벤트당 복사 제거는 되었지만, 커밋 1회의 비용은 개선하지 않아 여전히 프레임 시간 대비 오래 걸리는 현상 존재

### 남은 문제

커밋 **횟수**는 초당 10회로 고정, 커밋 **1회의 비용**은 여전히 로그 수(N)에 비례
flush마다 전체 배열 복사, 전체 로그 집계(buildHistogram), reverse, 전체 리스트 DOM 렌더가 일어나고,
배열과 DOM 노드는 상한 없이 계속 자라기 때문에 시간이 지날수록 커밋 1회가 무거워진다.

→ 다음 개선: 배열 크기 상한(메모리), 가상 스크롤(DOM)

## 개선 2: 차트 집계 매번 재계산 → 새로 추가되는 부분만 계산

`buildHistogram` 함수에서 flush마다 전체 로그를 처음부터 다시 순회하는 것을 발견
이미 집계된 로그를 다시 처리하는 것은 불필요한 접근인 것을 확인하여
매번 N건 순회하던 내용을 새로 들어온 batch만 Map에 더하고,
60초가 지난 칸은 지우는 식으로 로직 변경

> 문제 파악과 설계 방향은 직접 진행하였으나, 구현은 AI 활용 후 성능 검증 직접 수행

### 결과

- Performance 패널 프로파일(Bottom-Up)에서 `buildHistogram`이 검색 결과 없음(실행 안 됨)으로 확인
- 새로 추가된 `addBatchToLedger`/`buildBars`도 확인되지 않음
- Scripting 비율 (×10 기준): **85% → 30.5%** (1,703ms / 5.59s)

## 개선 3: react-window로 로그 리스트 가상화

로그가 N건이면 DOM에도 N행을 전부 생성중
react-window의 List로 교체하여 보이는 행(+overscan)만 렌더하도록 변경
행 어댑터가 배열을 뒤에서부터 인덱스로 읽게 하면서,
기존의 `[...logs].reverse()` (매 flush마다 N건 복사)도 함께 제거

### 결과

- 부하 ×100도 버벅임 없이 소화 가능한 수준
- 로그를 빠르게 스크롤하는 경우 일시적인 DOM 노드 증가는 존재 (overscan에 의한 정상 동작)

### 남은 문제

- 오랜 시간 페이지에 머무르면 logs 배열이 계속 무거워지고, js heap size가 증가
