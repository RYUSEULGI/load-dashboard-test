## 문제: 트래픽 10배에 프레임 시간 23배

평상시(50건/초)에는 프레임이 16.7ms안에 그려지지만,
부하를 10배(500건/초)로 올리면 프레임당 340~390ms, 약 2.7fps로 떨어진다.

Performance 측정 결과 병목은 Painting(187ms)이 아니라 **Scripting(3,292ms / 3.87s, 85%)**

| 지표           | 평소 (50/s) | ×10 (500/s)         |
| -------------- | ----------- | ------------------- |
| 프레임 시간    | ≤16.7ms     | 340~390ms (약 23배) |
| Scripting 비율 | 47%         | 85%                 |
| DOM 노드       | 17,581 ↑    | 63,268 ↑            |
| JS heap        | 164MB ↑     | 238MB ↑             |

> 측정 조건: Chrome DevTools Performance, 각 3~5초 녹화, 동일 조건에서 비교

- DOM 노드와 heap이 평상시에도 계속 증가
- 로그 1건이 DOM 노드 7개를 만들고 배열은 계속 증가하기 때문에, 부하가 없어도 시간이 지나면 느려지는 건 동일하게 발생

#### 파악된 문제

- 이벤트마다 setState 호출
- 전체 로그를 매번 DOM에 렌더
- 매 렌더마다 전체 배열 순회(집계·reverse).

---

## 개선 1: 이벤트당 setState → 100ms 배치 flush

이벤트가 올 때마다 setState 하던 것을, useRef 큐에 모아뒀다가
100ms 간격으로 한 번에 커밋하도록 변경했다.

- 이벤트 도착: `waitingLineRef.current.push(e)` — 리렌더를 일으키지 않는 enqueue (상각 O(1))
- 100ms마다: 큐를 통째로 꺼내고(swap) 새 빈 큐로 교체 → 한 번의 setState로 커밋
- updater 안에서 ref를 읽지 않고 batch를 먼저 캡처 — updater는 순수 함수여야 하므로 (StrictMode 이중 실행 안전)

**100ms인 이유**: RAIL 모델 기준 100ms 이내 반응은 즉각적으로 인지되므로, 로그가 최대 100ms 늦게 보이는 지연을 내주는 대신 커밋 횟수를 최대 5,000회/초 → 10회/초로 제한하는 트레이드오프.

### 결과

| 지표 | 개선 전 | 개선 후 |
| --- | --- | --- |
| Scripting 비율 (평소) | 47% | **31%** (1,671ms / 5.41s) |
| CPU 패턴 (평소) | 연속 포화 | 100ms 주기 펄스 |
| Long Task (평소) | 상시 발생 | 거의 소멸 |
| 프레임 시간 (×10) | 340~390ms | (동일 조건 재측정 예정) |

### 남은 문제

×10에서는 여전히 Scripting 비율이 84%로 높다. 커밋 **횟수**는 10회/초로 제한했지만,
커밋 1회의 **비용**이 여전히 O(n)이기 때문이다 — 매 flush마다 전체 배열 순회(집계·reverse)와
전체 리스트 DOM reconciliation(8만+ 노드)이 일어난다. 그리고 배열과 DOM은 여전히 상한 없이 자란다.
